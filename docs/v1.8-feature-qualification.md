This record captures the release decision for the five capabilities introduced between v1.7.0 and the v1.8.0 release candidate. The supported scope below is the only scope affected by graduation. Each deferred expansion remains preview work and must receive its own feature flag when it is implemented.

## Stable scope and rollback

The two graduated features are enabled in `dev` and `release` channels by default:

- `python-runtime-capture` (`LOP-FEAT-0014`)
- `python-codemod-suggestions` (`LOP-FEAT-0016`)

Their former `-preview` names remain deprecated aliases for compatibility. An operator can explicitly disable either stable feature by canonical name or code. No v1.8 release lock is required.

Python runtime capture remains command-explicit: Lopper never launches a test command unless the user supplies `--runtime-test-command`. Explicit trace consumption remains controlled independently by `python-runtime-trace`; disabling that feature cannot be bypassed merely because capture is stable.

Python codemods remain syntactically conservative rather than a proof that an import has no module side effects. Mutation still requires `--apply-codemod --apply-codemod-confirm`, refuses a dirty worktree by default, and writes rollback evidence.

## Preview boundaries

The three retained preview features stay disabled by default in stable release builds. Their current limitations must remain visible in user documentation:

- CycloneDX output is direct-dependency-only and does not infer versions or PURLs.
- Vulnerability matching uses package name and ecosystem; OSV affected-version ranges are not yet evaluated, and `fixedVersion` is informational.
- The dashboard remediation queue can only classify evidence present in dashboard job reports until dashboard/analyse policy parity lands.

Graduating the two supported v1.8 scopes does not implicitly graduate any follow-on issue listed above.
